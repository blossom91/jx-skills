# TAPD 使用规范（平台中心）

## 1. 创建需求时的必填字段

创建需求（story）时，以下字段**必须填写或提醒用户提供**：

| 字段 | 类型 | 规则 |
|------|------|------|
| 名称 | text | 应体现可交付的功能范围 |
| 处理人 | user | **唯一**，不可多人 |
| 优先级 | select | P0/P1/P2/P3，四选一 |
| 分类 | select | 单选，各项目空间有不同候选值 |
| 需求类型 | select | 最新功能/功能迭代/体验优化/技术改造 |
| 需求来源 | multi_select | 可多选，从部门列表中选 |
| 描述 | text | **必须包含成本信息**（见下方） |

### 描述中必须包含的成本信息

1. **人力成本** — 产研工时
2. **资源成本** — 如有新增资源（如服务器、云资源等）
3. **其他成本** — 如有采购/行政参与

> 如果用户未提供成本信息，应主动提醒补充。

---

## 2. 需求颗粒度

- 单个需求应为**一个迭代可全量交付**的内容
- 分多期的需求应拆为多个独立需求（一期、二期各建一个）
- 不应将多期混在一个需求中

---

## 3. 子任务拆解规范

### 层级结构

```
需求（顶层）
├── 产品任务（汇总）     ← 第二层：每个职能仅一条
│   ├── 子任务 A        ← 第三层：可按复杂度拆分
│   └── 子任务 B
├── 设计任务（汇总）
├── 后端任务（汇总）
│   ├── 接口开发
│   └── 数据迁移
├── 前端任务（汇总）
├── 测试任务（汇总）
└── 运维任务（汇总）
```

### 规则

1. **第二层级每个职能仅一条汇总工作项**，不要重复创建
2. 使用对应的 **workitem_type_id** 创建（不要全用通用任务）
3. 层级不限，但通常**平铺即可**
4. 上层节点处理人：负责统筹，保证拆解和工时合理性
5. **末尾节点（叶子节点）必须填写**：
   - 预计开始（begin）
   - 预计结束（due）
   - 预估工时（size）

### 工时约束

- 最小统计单位：**0.5 小时**
- 单个事项控制在：**0.5 ~ 4 小时**
- 超过 4 小时应继续拆分子任务

### 字段继承

子任务会**自动继承**父需求的以下字段：
- 优先级
- 分类
- 迭代

> 创建子任务时无需重复填写这些字段，API 会自动继承。父项的工时字段由子项自动汇总。

---

## 4. 迭代关联（强制）

- **所有发布上线的需求和缺陷必须关联迭代**
- 禁止未关联迭代的需求或缺陷发布上线
- 无法在当前迭代交付的应及时移出迭代
- 已结束的迭代应及时关闭，未完成的移入后续迭代
- 迭代时间应**连续且无交集**

> 创建需求/缺陷时，如果用户未指定迭代，应主动提醒关联。

---

## 5. 缺陷管理规范

### 创建缺陷必填字段

| 字段 | 规则 |
|------|------|
| 标题 | 清晰描述缺陷现象 |
| 处理人（current_owner） | **唯一**，通常为研发或产品 |
| 验证人（confirmer） | **必填**，通常为 QA，是缺陷跟进的第一责任人 |
| 优先级 | P0/P1/P2/P3 |
| 缺陷类型（bugtype） | 功能缺陷/性能缺陷/安全漏洞/兼容性缺陷/其他 |
| 严重性（custom_field） | 致命/严重/一般/轻微/建议 |

### 缺陷关联规则

- 测试/验收中发现的缺陷**必须关联到顶层需求**（不是子任务）
- 线上问题和用户反馈除外（可不关联需求）

### 严重性定义

| 等级 | 定义 |
|------|------|
| 致命 | 系统崩溃、数据丢失、核心功能完全不可用 |
| 严重 | 核心功能异常但有绕过方案，影响主流程 |
| 一般 | 非核心功能异常，不影响主流程 |
| 轻微 | UI 样式、文案错误等 |
| 建议 | 体验优化建议，非缺陷 |

---

## 6. 状态更新规范

- **每日下班前**：更新当日推进事项的状态
- **每日上班后**：查看自己作为处理人的事项，规划当日待办
- 验证人需确保**每日**及时更新缺陷状态

---

## 7. 优先级定义

| 级别 | 含义 |
|------|------|
| P0 | 紧急，阻塞性问题，需立即处理 |
| P1 | 高优，当前迭代必须完成 |
| P2 | 中优，可延后但需规划 |
| P3 | 低优，有空再处理 |
